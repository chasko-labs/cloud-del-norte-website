## Naming Standard Established

### Decision: Translation Key Naming Standard (DEC-009 candidate)

**Pattern:** `{semanticContext}.{specificContent}`

**Principles:**
1. **Semantic over location** — Key describes content, not page
2. **Component context first** — Group by UI component or content domain
3. **Nested structure** — Use dot-notation for hierarchy
4. **Reusability** — Keys portable across pages if content shared
5. **Self-documenting** — Key name hints at rendered content

**When to use page prefix:**
- Content truly page-specific (breadcrumb, page header)
- Content would conflict if moved to another page
- Page name is semantically meaningful (Dashboard, not Home)

**Examples:**
- ✅ `dashboardPage.header` (page-specific breadcrumb)
- ✅ `userGroupHero.description` (component-specific, portable)
- ✅ `pieChart.chartAriaRoleDescription` (global, reusable)
- ❌ `home.header` (generic, no semantic value)
- ❌ `home.infoLink` (vague, requires code inspection)
